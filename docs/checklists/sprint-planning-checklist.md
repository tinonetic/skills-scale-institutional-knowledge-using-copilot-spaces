# Sprint Planning Checklist

Use this checklist to ensure effective and thorough sprint planning sessions.

---

## Pre-Planning (Before the Meeting)

### Product Manager / Product Owner

- [ ] Backlog is prioritized and up to date
- [ ] Top priority items have clear acceptance criteria
- [ ] User stories are sized (T-shirt or story points)
- [ ] Dependencies are identified and documented
- [ ] Sprint goal is drafted
- [ ] Stakeholder feedback has been incorporated

### Project Manager

- [ ] Sprint planning meeting scheduled with all required attendees
- [ ] Previous sprint's velocity and completion rate reviewed
- [ ] Team capacity confirmed (account for PTO, holidays, non-sprint work)
- [ ] Previous sprint's action items reviewed
- [ ] Sprint planning agenda circulated

### Development Team

- [ ] Previous sprint retrospective action items reviewed
- [ ] Technical dependencies and blockers identified
- [ ] Design/architecture decisions documented for complex stories
- [ ] Questions about upcoming work submitted in advance

---

## During Planning (The Meeting)

### Review & Context (15-20 minutes)

- [ ] Review sprint goal and objectives
- [ ] Review previous sprint outcomes and velocity
- [ ] Discuss any changes in priorities or team capacity
- [ ] Highlight dependencies or risks

### Backlog Refinement & Commitment (30-45 minutes)

- [ ] Walk through prioritized backlog items
- [ ] Clarify acceptance criteria for each story
- [ ] Break down large stories if needed
- [ ] Identify technical tasks and subtasks
- [ ] Confirm estimates or re-estimate as needed
- [ ] Team commits to sprint backlog based on capacity
- [ ] Definition of Done reviewed and confirmed

### Task Assignment & Planning (15-20 minutes)

- [ ] Stories assigned to team members (or pulled during sprint)
- [ ] Initial tasks created and assigned
- [ ] Cross-team dependencies documented and communicated
- [ ] Clarify who will handle code reviews, testing, documentation

### Finalize Sprint Plan (10 minutes)

- [ ] Sprint goal confirmed and documented
- [ ] Sprint backlog locked in project board
- [ ] Key milestones or demo dates confirmed
- [ ] Risks and blockers captured in risk register
- [ ] Next standup scheduled

---

## Post-Planning (After the Meeting)

### Project Manager

- [ ] Sprint board updated with committed work
- [ ] Sprint goal and summary shared with stakeholders
- [ ] Dependencies communicated to relevant teams
- [ ] Sprint burndown chart or tracking dashboard set up
- [ ] Meeting notes and action items documented

### Development Team

- [ ] All committed stories are in the "Ready" column
- [ ] Development environment prepared (branches, feature flags, etc.)
- [ ] First stories pulled into "In Progress"
- [ ] Any pre-work or setup tasks completed

### Product Manager

- [ ] Sprint goal and scope communicated to stakeholders
- [ ] Acceptance criteria for top-priority items available for reference
- [ ] Ready to support team with clarifications during sprint

---

## Sprint Planning Anti-Patterns to Avoid

- [ ] **Avoid:** Planning without a clear sprint goal
- [ ] **Avoid:** Overcommitting based on ideal capacity (account for interruptions)
- [ ] **Avoid:** Including stories without clear acceptance criteria
- [ ] **Avoid:** Skipping estimation or technical feasibility discussion
- [ ] **Avoid:** Ignoring dependencies or assuming they'll resolve themselves
- [ ] **Avoid:** Planning sessions that exceed 2 hours (for a 2-week sprint)

---

## Tips for Effective Sprint Planning

1. **Timeboxing**: Keep planning sessions to 1-2 hours max for a 2-week sprint
2. **Right Sizing**: If a story can't be completed in the sprint, break it down
3. **Buffer Time**: Reserve 10-20% capacity for unplanned work and bug fixes
4. **Collaborative Estimation**: Use team-based estimation (planning poker, etc.)
5. **Clear Exit Criteria**: Don't leave planning until the sprint goal and scope are crystal clear
